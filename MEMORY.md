# 📋 开发规范（必须遵守）

## 🚨 血泪教训（2026-03-25）

### 违规案例 1：Git 流程违规（21:13）
- **问题：** 直接在 main 分支开发和推送
- **教训：** 推送前必须 `git branch` 确认分支

### 违规案例 2：文档同步违规（21:17）
- **问题：** 提交代码时没有同步更新文档
- **教训：** 代码和文档必须一起提交

### 违规案例 3：事后创建分支（21:20）
- **问题：** 推送后才创建 feature 分支
- **教训：** 分支必须在开发前创建

### 违规案例 4：Flask 重构未测试先行（21:47）
- **问题：** 26 文件重构，未 TDD，PR 过大
- **教训：** 测试先行不是事后补，大重构必须拆分 PR

### 违规案例 5：文档更新未走 PR（22:37）
- **问题：** Sprint 6 计划文档直接推送到 main，未创建 feature 分支和 PR
- **原因：** 觉得"只是文档"就放松了要求
- **教训：** 规范是铁律，不应该有例外（无论代码还是文档）

### 违规案例 6：重构未更新外部配置（2026-03-26 13:32）
- **问题：** Sprint 5 Flask 重构时脚本移至 `scripts/` 目录，但 crontab 路径未同步更新，导致推送失败约 24 小时
- **原因：** 
  1. 重构只关注代码本身，忽略了外部配置（crontab、systemd 等）
  2. Crontab 配置不在 Git 版本控制中，无法通过 code review 发现
  3. 缺少推送失败告警机制，依赖用户主动发现
- **影响：** 股票监控推送中断约 24 小时，损失 20-30 次推送
- **教训：** 
  - **配置即代码** - 所有配置都应纳入 Git 管理
  - **重构要完整** - 代码移动后必须检查所有引用点（配置文件、部署脚本、文档、监控）
  - **监控需先行** - 自动化任务必须有执行日志和失败告警

- ❌ 禁止直接在 main 分支开发（**22:37 又犯！**）

- ❌ 禁止未测试先行就开发
- ❌ 禁止单 PR 过大（>10 文件需拆分）
- ❌ 禁止提交后不同步文档
- ❌ 禁止重构后不更新外部配置（**2026-03-26 新增！**）

## 🔄 开发流程（12 步）

```
1. 文档先行          → AGILE_DEVELOPMENT.md 添加 Sprint 记录
2. 创建 Git 分支      → git checkout -b feature/xxx
3. 编写测试用例      → pytest 测试（预期失败）
4. 运行测试（红）     → 确认测试失败
5. 代码开发          → 在功能分支上编写
6. 运行测试（绿）     → 确认测试通过
7. 重构优化          → 保持测试通过
8. 功能测试          → 单元/集成/数据验证
9. 检查 PR 拆分      → >10 文件？拆分多个 PR
10. 提交分支         → git push -u origin feature/xxx
11. PR 审查          → GitHub 创建 Pull Request
12. 同步更新文档     → README/ROADMAP/AGILE/TESTING
```

```bash

# ✅ AGILE/README/ROADMAP 更新了吗？

# PR 检查（大型重构）
# ✅ 变更文件<10 个？否→拆分 PR
```

## 🔄 重构完整性检查清单（2026-03-26 新增）

**重构代码后必须检查：**

```bash
# 1. 配置文件
grep -r "旧路径" config/ scripts/ *.sh 2>/dev/null
crontab -l | grep "旧路径"
systemctl status *.service 2>/dev/null

# 2. 部署脚本
grep -r "旧路径" deploy/ .github/ scripts/ 2>/dev/null

# 3. 文档引用
grep -r "旧路径" *.md docs/ 2>/dev/null

# 4. 监控告警
# ✅ 日志路径是否正确？
# ✅ 告警规则是否需要更新？
```

**检查项：**
- [ ] Crontab 配置已更新
- [ ] Systemd 服务文件已更新（如有）
- [ ] 部署脚本已更新
- [ ] 文档引用已更新
- [ ] 监控告警规则已更新
- [ ] 日志路径已验证

## 📝 文档同步规范

| 文档 | 更新内容 | 时机 |
|------|---------|------|
| `AGILE_DEVELOPMENT.md` | Sprint 记录、测试指标 | 代码提交前 |
| `README.md` | 功能说明、使用方法 | 代码提交前 |
| `ROADMAP.md` | 版本发布记录 | 代码提交前 |
| `MEMORY.md` | 违规案例、经验教训 | 事件复盘后 |

**Commit Message 格式：** `<type>: <description>`

类型：`feat` `fix` `docs` `refactor` `test` `chore`

## 💡 核心原则

**文档先行 → 测试先行 → 分支开发 → 小步提交 → PR 审查 → 同步更新**

**记住：**
- 规范不是束缚，是保证质量的工具
- 走捷径最终会走更远的路
- 每次违规都是学习机会
- 自律才能自由
- 测试先行不是事后补（TDD 是红 - 绿 - 重构循环）
- 大重构必须拆分 PR（便于审查和回滚）
- **配置即代码，重构要完整，监控需先行**（2026-03-26 新增）
